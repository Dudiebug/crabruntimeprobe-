# CrabRuntimeProbe Field Guide

CrabRuntimeProbe is a standalone UE4SS Lua diagnostic and research mod for Crab Champions. Its job is to gather runtime evidence about which reads are safe in specific contexts, roles, lifecycle states, and access paths.

CrabRuntimeProbe is not CrabInvSync, not CrabSyncV2, and not a sync implementation. It must stay non-mutating: no gameplay state writes, no mutating RPC calls, no joined-client apply behavior, and no inventory rebuild logic.

## Why Objectdump Is Not Enough

Objectdump facts prove that a class, struct, property, or function name exists in static dump data. They do not prove that UE4SS Lua can safely read that symbol during startup, menu, lobby, solo, host, joined-client, travel, respawn, or replication.

Runtime-confirmed safe access means a probe ran in-game, through a specific access method, under recorded gates and context, and produced evidence. Safety is scoped to that exact access path. `CrabPS.WeaponDA` through `GetPropertyValue` is not proof that direct field access, nested traversal, or writes are safe.

## Normal Testing Workflow

1. Run a prepare script from the real Git checkout, usually one of the `quick-*-prepare.ps1` helpers.
2. Launch Crab Champions manually.
3. Wait in the requested context from the prepare script or README: menu, solo run, host lobby, multiplayer run, or stable local world.
4. Quit the game.
5. Run the paired collect script.
6. Import evidence with `scripts/import-latest-runtime-evidence.ps1` when the pass should be preserved in the repo.
7. Regenerate docs/wiki staging through the import script, or run the documented Node generation tools only when that is the intended docs pass.

The repo helpers prefer the default Steam path:

```powershell
C:\Program Files (x86)\Steam\steamapps\common\Crab Champions\CrabChampions\Binaries\Win64
```

For a custom install, use `scripts/run-local-diagnostic-cycle.ps1 -GameBin "<path>"` with the matching prepare or collect switch.

## Runtime Artifacts

`probe_results_<session>.jsonl` records probe rows and observe rows from the Lua runtime. Observe rows are passive `Observe.Context` snapshots; active rows are curated registry probes after pacing and gates.

`access_evidence_<session>.jsonl` records normalized evidence rows used by the import and docs pipeline. These rows include symbol, access method, runtime status, context, role, lifecycle state, and safety gates.

`session_manifest_<session>.json` records the session id, build info, config snapshot, probe set, tick driver, safety gates, and active research gates.

`diagnostic_summary.txt` is generated by collection scripts from installed game artifacts, UE4SS logs, manifests, and JSONL checks. It is the quick human-readable upload/debug summary.

`UE4SS.log` is the game-side log. It is needed for startup diagnostics, tick driver confirmation, session id/commit matching, and crash breadcrumb inference.

## Generated Docs

`docs/RUNTIME_EVIDENCE_INDEX.md` is the high-level imported evidence summary and should be treated as the current runtime proof index.

`docs/SAFE_ACCESS_MATRIX.md` lists SAFE, RETURNS_NIL, and other runtime statuses scoped to access method, context, role, lifecycle, and session evidence.

`docs/PROBE_CANDIDATES.md` is objectdump/candidate planning. It is not proof by itself.

`docs/PROBE_RESULTS.md` is a generated probe status summary from result rows and may be sparse.

`docs/CRASH_PHASE_SUMMARY.md` is generated crash breadcrumb analysis and may be sparse when no crash suspect is inferred.

## Observe Mode And Active Mode

`observe` mode is passive. It writes context rows after `startupWarmupTicks`, every `observeIntervalTicks`, and does not run the curated probe registry.

`active` mode runs one curated probe at a time after startup warmup, context stability, interval pacing, maximum-probe limits, and safety-gate checks.

## Safety Gate Model

`allowHudTickHook` controls the HUD `ReceiveDrawHUD` tick driver. It is restricted because the HUD hook has crash history.

`allowDeepArrayProbes` gates risky nested array traversal and element dereference probes.

`allowInventoryInfoProbes` gates reads of `InventoryInfo` and nested inventory metadata.

`allowHealthProbes` gates explicit health reads.

`allowIdentityProbes` gates identity and roster probes.

`allowRawIdentityEvidence` controls raw identity output. It defaults false; normal evidence should remain fingerprinted/redacted.

`allowResourceVisibilityProbes` gates resource visibility probes across visible PlayerStates.

`allowInventoryArrayShallowProbes` gates shallow local inventory array shape/count research.

`allowInventoryArrayCountProbes` gates the narrow local wrapper count metadata phase. It does not permit traversal or element reads.

`allowInventoryElementDataAssetReadProbes` gates the narrow capped first-element identity phase. It does not permit full traversal, multiple elements, `InventoryInfo`, Enhancements, Level, AccumulatedBuff, gameplay function calls, writes, RPCs, HUD hooks, or broad deep arrays.

`allowWriteProbes` gates write probes. RuntimeProbe must keep this false for normal work and should not grow mutating probes.

`allowRpcProbes` gates RPC-related probes. RuntimeProbe may document function presence, but must not call mutating RPCs.

The current config also includes narrower gates such as `allowInventoryArrayShapeConfirmProbes`, `allowInventoryUserdataIntrospectionProbes`, `allowUnknownRoleProbes`, and `allowJoinedClientDeepProbes`; they reinforce the same rule: prove each parent layer before going deeper.

## Restricted Areas

The HUD tick hook is restricted because local evidence shows immediate crash risk in this setup.

Deep array traversal, full inventory element traversal, `InventoryInfo`, and `Enhancements` remain unproven unless current imported evidence later proves them.

Writes and mutating RPCs are outside RuntimeProbe's purpose.

Joined-client apply behavior is future CrabSyncV2 work and must not be treated as RuntimeProbe behavior.

## Which Script Do I Run Next?

Use `scripts/quick-smoke-prepare.ps1`, launch to menu for 20 to 30 seconds, quit, then use `scripts/quick-smoke-collect.ps1` for the first smoke pass.

After smoke, use `scripts/quick-tickdriver-prepare.ps1 -TickDriver executeDelay`, launch to menu, quit, then `scripts/quick-tickdriver-collect.ps1`.

For passive gameplay context, use `scripts/quick-gameplay-observe-prepare.ps1`, play or host in a stable context for 30 to 60 seconds, quit, then collect.

For current read-only research phases, prefer the named quick scripts in README order or the campaign pair: `scripts/quick-campaign-prepare.ps1`, manual run, then `scripts/quick-campaign-collect.ps1`.

After a successful pass that should be preserved, run `scripts/import-latest-runtime-evidence.ps1`.

Prefer `install-client-to-game.ps1` or `export-client-folder.ps1` over drag-and-drop stale folders. The helpers validate safe defaults and stamp build info.

## How To Interpret A Result

`SAFE` means safe only for the recorded symbol, access method, context, role, lifecycle state, and gates. It is not a blanket permission.

`RETURNS_NIL` is not a crash. It usually means the path was absent, unavailable, or not exposed in that context.

Userdata shape visibility is not traversal proof. Seeing `userdata` for inventory arrays does not prove safe count, element wrappers, element dereference, item fields, or sync.

Count metadata is not item metadata proof. A length or count result, if present, still does not prove DA identity, `InventoryInfo`, `Enhancements`, or item preservation.

Inventory element DA evidence is capped identity evidence only. One first-element result does not prove full traversal, multiple element reads, InventoryInfo, Enhancements, Level, AccumulatedBuff, or item synchronization.

`pcall` catches Lua errors. It cannot guarantee protection from native access violations, stale pointers, invalid UObject state, or UE4SS/native crashes.
